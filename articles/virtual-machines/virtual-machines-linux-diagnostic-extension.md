Collecte et charge les données de performances, de diagnostic et de syslog du système des machines virtuelles Linux dans la table de stockage de l’utilisateur.2.	Permet à utilisateur de personnaliser les métriques des données collectées et chargées.	3.	Permet à l’utilisateur de charger les fichiers journaux spécifiés dans la table de stockage désignée.Pour la version 2.0, les données comprennent :1.	Toutes les journalisations de Rsyslog de Linux, y compris les journaux système, de sécurité et des applications.2.	Toutes les données système spécifiées dans ce [document](https://scx.codeplex.com/wikipage?title=xplatproviders&referringTitle=Documentation "document").3.	Les fichiers journaux spécifiés par l’utilisateur.##Comment activer l’extension L’extension peut être activée via le portail de gestion Azure, Azure PowerShell ou des scripts CLI Azure.Pour afficher et configurer les données système et de performances directement à partir du [portail de gestion Azure](https://ms.portal.azure.com/#), suivez ces [étapes](http://azure.microsoft.com/blog/2014/09/02/windows-azure-virtual-machine-monitoring-with-wad-extension/ "URL vers le blog de Windows"). Cet article est centré sur l’activation et la configuration de l’extension via des commandes CLI Azure. Ceci vous permet de lire et d’afficher directement les données de la table de stockage.##Composants requis1. Agent Microsoft Azure Linux version 2.0.6 ou ultérieure. Notez que la plupart des images de la galerie Linux de machines virtuelles Azure comprenaient la version 2.0.6 ci-dessus. Vous pouvez exécuter WAAgent -version pour vérifier la version installée dans la machine virtuelle. Si la machine virtuelle exécute une version antérieure à 2.0.6, vous pouvez suivre ces [instructions](https://github.com/Azure/WALinuxAgent "instructions") pour la mettre à jour.2.  [Interface CLI d’Azure](./xplat-cli.md "Interface de ligne de commande Azure") : suivez [ce guide](./xplat-cli-install.md "ce guide") pour configurer l’environnement CLI d’Azure sur votre ordinateur. Une fois que l’interface CLI d’Azure a été installée, vous pouvez utiliser la commande Azure à partir de votre interface de ligne de commande (Bash, Terminal, Invite de commandes) pour accéder aux commandes CLI d’Azure. Par exemple, exécutez « azure vm extension set --help » pour une utilisation détaillée, exécutez « azure login » pour vous connecter à Azure, ou exécutez « azure vm list » pour répertorier toutes les machines virtuelles que vous avez sur Azure.3. Compte de stockage pour stocker les données. Vous devez disposer d’un nom de compte de stockage et d’une clé d’accès préalablement créés pour charger les données dans votre stockage.##Utiliser une commande CLI Azure pour activer l’extension de diagnostic Linux ###  Scénario 1 Activer l’extension avec le jeu de données par défautPour la version 2.0 et ultérieure, les données collectées par défaut comprennent : 1. Toutes les informations de Rsyslog (y compris les journaux système, de sécurité et des applications)  2. Un ensemble principal de données système de base. Notez que le jeu de données complet est décrit dans ce [document](https://scx.codeplex.com/wikipage?title=xplatproviders&referringTitle=Documentation "document"). Si vous voulez activer des données supplémentaires, poursuivez avec les étapes des scénarios 2 et 3.Étape 1. Créez un fichier nommé PrivateConfig.json avec le contenu suivant :	{     	"storageAccountName":"the storage account to receive data",     	"storageAccountKey":"the key of the account"	}Étape 2. Exécutez « azure vm extension set nom_machine_virtuelle LinuxDiagnostic Microsoft.OSTCExtensions 2.* --private-config-path PrivateConfig.json ».###   Scénario 2 Personnaliser les métriques de surveillance des performances  Si vous voulez personnaliser la table des données de performances et de diagnostic, suivez les étapes ci-dessous.Étape 1. Créez un fichier nommé PrivateConfig.json avec le contenu suivant, spécifiez les données particulières que vous voulez collecter.Pour tous les fournisseurs et les variables pris en charge, consultez ce [document](https://scx.codeplex.com/wikipage?title=xplatproviders&referringTitle=Documentation "document"). Vous pouvez avoir plusieurs requêtes et les stocker dans plusieurs tables en ajoutant d’autres requêtes dans le script.Notez que par défaut, les données de Rsyslog sont toujours collectées.	{     	"storageAccountName":"storage account to receive data",     	"storageAccountKey":"key of the account",      	"perfCfg":[           	{"query":"SELECT PercentAvailableMemory, AvailableMemory, UsedMemory ,PercentUsedSwap FROM SCX_MemoryStatisticalInformation","table":"LinuxMemory"           	}             ]	}Étape 2. Exécutez « azure vm extension set nom_machine_virtuelle LinuxDiagnostic Microsoft.OSTCExtensions 2.* --private-config-path PrivateConfig.json »###   Scénario 3 Chargement de vos propres fichiers journauxSi vous voulez collecter des fichiers journaux particuliers et les charger dans votre compte de stockage, suivez les étapes ci-dessous. Vous devez spécifier le chemin d’accès à votre fichier journal, ainsi que le nom de la table où stocker vos journaux. Vous pouvez avoir plusieurs fichiers journaux en ajoutant plusieurs entrées fichier/table au script.Étape 1. Créez un fichier nommé PrivateConfig.json avec le contenu suivant :	{     	"storageAccountName":"the storage account to receive data",     	"storageAccountKey":"key of the account",      	"fileCfg":[           	{"file":"/var/log/mysql.err",             "table":"mysqlerr"           	}           ]	}Étape 2. Exécutez « azure vm extension set nom_machine_virtuelle LinuxDiagnostic Microsoft.OSTCExtensions 2.* --private-config-path PrivateConfig.json »###   Scénario 4 Désactiver l’extension de surveillance LinuxÉtape 1. Créez un fichier nommé PrivateConfig.json avec le contenu suivant :	{     	"storageAccountName":"the storage account to receive data",     	"storageAccountKey":"the key of the account",     	“perfCfg”:[],     	“enableSyslog”:”False”	}Étape 2. Exécutez « azure vm extension set nom_machine_virtuelle LinuxDiagnostic Microsoft.OSTCExtensions 2.* --private-config-path PrivateConfig.json »##Passer en revue vos donnéesLes données de performances et de diagnostic sont stockées dans la table de stockage Azure. Consultez [cet article](./storage-ruby-how-to-use-table-storage.md "cet article") pour savoir comment accéder aux données de la table de stockage à l’aide de scripts CLI Azure.Vous pouvez aussi utiliser les outils d’interface utilisateur suivants pour accéder aux données :1.	Utilisez l’Explorateur de serveurs de Visual Studio. Accédez à votre compte de stockage. Au bout d’environ 5 minutes de fonctionnement de la machine virtuelle, vous devez normalement voir les tables par défaut. « LinuxCpu », « LinuxDisk », « LinuxMemory », « Linuxsyslog ». Double-cliquez sur le nom de la table pour afficher les données. 2.	Utilisez l’[Explorateur du stockage Azure](https://azurestorageexplorer.codeplex.com/ "Azure Storage Explorer") pour accéder aux données. ![image](./media/virtual-machines-linux-diagnostic-extension/no1.png)Si vous avez activé les options fileCfg ou perfCfg spécifiées dans les scénarios 2 et 3, vous pouvez également afficher les données autres que les données par défaut dans les outils ci-dessus.##Problèmes connus1. Pour la version 2.0, les informations de Rsyslog et le fichier journal spécifié par l’utilisateur sont accessibles seulement via un script.2. Pour la version 2.0, si vous avez activé l’extension de diagnostic Linux d’abord via un script, vous ne pouvez pas afficher les données à partir du portail. En revanche, si vous activez l’extension d’abord depuis le portail, les scripts fonctionnent encore.Nous prévoyons de résoudre les deux problèmes dans la prochaine version.
